动态链接和静态链接

发表于 2019-05-12 | 更新于 2019-05-13

写在前面：之前我们在一篇文章中提到过，从你写的源代码到你运行的可执行文件，一共有三个步骤。其中之一就是链接。我们经常会听到类似动态链接，静态链接这类术语。我通过查阅资料来简单回答一下这个问题。

  • 将文件静态链接到可执行文件时，该文件的内容会在链接时，与汇编代码一同被包含在可执行文件中。换句话说，文件的内容将插入到您将运行的可执行文件中，真正意义（physically ）上的插入。
  • 将文件动态链接到可执行文件时，该文件的内容不会真正的被包含在可执行文件中，取而代之的是指向该文件的指针（例如，文件的文件名）。只有在你真正运行可执行文件的时候才会被包含进来，而且他们只会被加载到内存中，而不是磁盘上。

从某种意义上说，动态链接推迟了链接这一步骤。还有一种更加推迟的办法，那就是只有当你调用函数的时候才会开始动态链接，而不是运行时。

静态链接的文件在链接时被“锁定”到可执行文件，因此它们无法改变（当然啦，除非你重新编译）。可执行文件引用的动态链接文件只需更换磁盘上的文件即可更改。这样的话我们无需重新链接代码，就可以更新（第三方库中的）功能；因为每次运行代码的时候，加载程序都会重新链接一遍。

这样做有好处也有坏处：

  • 更新和修复bug显得更加容易
  • 如果更新不兼容，就会使得程序崩溃

“DLL hell”说的就是这种情况：如果你用一个不兼容的库替换一个动态链接的库，所有有关的应用程序都可能会崩溃。

C++的编译过程

发表于 2019-05-12

写在前面：本文是How does the compilation/linking process work?高票回答的翻译。有能力的读者建议阅读原文。

C++的编译（compilation）过程包括三个步骤：

  1. 预处理（Preprocessing）：预处理器拿到C++的源码，处理类似#include，#define以及其他的预处理指令。这一部分的输出是所谓没有预处理指令的“纯C++文件”。
  2. 编译（compilation）：编译器拿到预处理器的输出并生成目标文件。
  3. 链接（Linking）：链接器拿到编译器产生的目标文件并生成库文件或者可执行文件。

预处理

预处理器需要解决的是类似#include以及#define这一类预处理指令。预处理指令与C++的语法无关，这也是必须谨慎使用它们的原因。

预处理器一次只能处理一个C++源文件。用相应文件中的内容（通常只是声明）去替换#include，宏替换（#define）以及根据#if, #ifdef 还有#ifndef来选择文本的不同部分。

预处理器利用一系列token来工作。（The preprocessor works on a stream of preprocessing tokens.）例如，宏替换被定义为用一个token替换另一个token（操作符##是合并两个token）。

之后，预处理器就根据上述的变换产生token流。它还添加了一些特殊的标记，告诉编译器每行代码的来源，以便编译器产生正确的错误消息。

编译

编译阶段是作用在预处理阶段的输出上。编译器解析纯C++源代码（没有任何的预处理指令）并将其转换为汇编代码。然后调用底层后端（工具链中的汇编器）将代码汇编成机器代码，生成某种格式的实际二进制文件（EOF，COFF，a,out，…）。此目标文件包含输入中定义的符号的已编译代码（以二进制形式）。目标文件中的符号通过名称来引用。（This object file contains the compiled code (in binary form) of the symbols defined in the input. Symbols in object files are referred to by name.）

目标文件可以引用未定义的符号，当你仅声明但没有定义的时候就是这样的情况。编译器不介意这一点，只要源代码格式正确，编译器就会愉快地生成目标文件。

我们可以单独编译每个源代码文件。这样的好处是，如果只更改一个文件，就无需重新编译所有的内容。

生成的目标文件可以放在称为静态库的特殊存档中，以便以后重用。

这个阶段就会产生我们工作中最常见的那一类编译错误，比如语法错误或重载解析错误等等。

链接

链接器从目标文件中生成最终的编译输出。这个输出可以是动态库（虽然名字很相似但是动态库和我们之前提到的静态库没有多少共同之处），也可以是可执行文件。

它通过用正确的地址替换未定义的符号引用来链接所有目标文件。这些符号可以在其他目标文件或者是库中（如果不是标准库就要告知链接器）定义。

这一阶段最常见的错误是缺少定义或重复定义。前者意味着定义不存在（即它们没有被写入），或者定义所在目标文件或库没有被告知链接器。后者很明显：在两个不同的目标文件或库中定义了相同的符号。

deltype

deltype是一个操作符，可以推导出括号内表达式的类型，其规则如下：

  1. 如果表达式e是一个变量，那么就是这个变量的类型。
  2. 如果表达式e是一个函数，那么就是这个函数返回值的类型。
  3. 如果不符合1和2，如果e是左值，类型为T，那么decltype(e)是T&；如果是右值，则是T。

合理利用deltype十分省事。

还有一个适合的用法是用来typedef函数指针，也会省很多事。

auto与deltype有一些subtle。说实话我一时半会无法牢牢记住。

nullptr

C/C++的NULL宏是个被有很多潜在BUG的宏。因为有的库把其定义成整数0，有的定义成 (void*)0。nullptr是强类型的。

右值引用和move语义

临时性变量（称为右值“R-values”）无法被改变，在C中亦同（且被视为等同于const T&）。

C++03低性能问题的之一，就是因为在值传递的时候发生不必要的深度拷贝。举个例子，如果一个函数要返回std::vector<T>，会把临时数据全部复制过去。然后临时对象会被析构，内存被释放。但是在c++ 11中我们可以把一个vector的右值引用作为std::vector的“move构造函数”，这样右值参数绑定的vector内部指向C-style数组的指针会直接复制给新的vector，然后该指针置为nullptr(活学活用🤭)。

这是一个经典的操作系统问题：什么是进程（process）？维基百科)说：进程是计算机中已运行程序的实体。我觉得这依旧是一个抽象的回答，我依旧无法在脑中将进程与现实里的任何事物对应。

后来我看到了某一种解释：

进程和线程都是一个时间段的描述，是CPU工作时间段的描述。

我觉得十分有收获。或许我们一直不应该从“空间”的角度去理解进程，它不是任何所谓“实体”，而仅仅只是对CPU工作的一段时间的描述罢了。

虚函数与多态

发表于 2018-09-05 | 更新于 2018-09-19

多态（polymorphism)一词源自希腊语，意为多种形态，多种形状。

举个例子，同样的一个求面积公式，对正方形而言，其形式为边长^2^；对圆而言，其形式为π半径^2^。“求面积”这一动作，在不同的几何体上有不同的形态。这就是我理解的多态。

譬如，有一个名叫Shape的类，分别派生出Rectangle类以及Triangle类。

可以看到它们各自都有一个area()函数。这些函数的函数体是不同的。在这个时候，我们仍然没有使用虚函数的必要。

为什么非要虚函数

可是，假如我们有这样一个需求。我们需要一个数组，一个可以同时容纳Rectangle以及Triangle的数组，我们该怎么办？无论是vector还是C中的数组，都要求元素类型相同。所以，换一种思路，我们可以容纳Shape的指针。这样，通过指针指向具体的派生类就可以间接实现我们的目标。看上去很完美。但是倘若我们要计算数组内几何体的面积，我们该怎么办？我们只有一系列的Shape指针，我们不知道这些指针是指向Rectangle还是Triangle，如果我们直接通过Shape指针调用area(),尴尬的地方在于它只会调用Shape类的area()。这是因为编译器想当然地认为Shape指针指向的肯定是Shape类，所以自然而然地就会调用Shape类的area()。

这个时候，虚函数粉墨登场了。只需要在Shape类中的area()函数定义之前加上virtual关键字，area()就成为了虚函数。

再回过头来看，就会发现，编译器很聪明地知道了Shape指针指向的是Rectangle类还是Triangle，随即调用相应地area()。

虚函数的实现机制

在C++中，虚函数的实现机制是虚函数表以及虚指针。

具体而言，每一个拥有虚函数的类及其子类都有一张虚函数表，该虚函数表为该些类通用；每一个该些类的对象都一个虚指针指向这张表，该虚指针为该对象独有。

因为class A是有虚函数的，所以编译器为其准备了一张虚函数表vtableA。由于h()不是虚函数，所以不在表内。

当基类指针指向派生类的时候，

编译器仍然认为pa是指向class A的指针，pa只能访问b的前两项，也就是继承自class A的那一部分。当执行pa->g(); 的时候，编译器还是“笨笨”地调用虚函数表中的第二项，而此时虚函数表中的第二项以及成为了B::g()的函数地址，所以就会调用B::g()，就这样实现了虚函数。

总结

对虚函数我们仍然只有一个感性的认识，我们还是有很多不了解的地方。譬如类的虚函数表是放在哪里的？对象的虚指针放在哪里？虚指针的空间效率问题等等。但是有了这样一个感性的认识，对接下里的深入理解会有很大的帮助。（虽然更多的是心理因素O(∩_∩)O）
